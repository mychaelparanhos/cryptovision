**Partitioning approach:**
- Supabase Postgres supports native declarative partitioning
- Use `CREATE TABLE ... PARTITION BY RANGE (timestamp)`
- Create child tables per month: `funding_rates_2026_04`, `funding_rates_2026_05`, etc.
- Drizzle schema.ts stays unchanged (it defines the logical table)
- Migration is raw SQL, not Drizzle-generated

**Publisher aggregation logic:**
- Funding rate: weighted average across exchanges (weight = OI of that exchange for that symbol)
- OI: sum across exchanges
- Liquidations: union of recent events across exchanges, sorted by timestamp
