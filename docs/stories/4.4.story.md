# S2C-4: Telegram Bot + Rich Alerts

**Epic:** Sprint 2 Expandida — Parte C (Entrega do Valor Pago)
**Status:** Draft
**Type:** Backend (Alerts Worker)
**Effort:** L (1.5 days)
**Priority:** P0 (core paid feature)
**Depends on:** Alert rules schema (existing), Redis PUB/SUB (existing)

## Story

Como usuario pago, quero receber alertas ricos no Telegram quando minhas regras forem acionadas (funding extremo, liquidacoes grandes, mudancas de OI), para agir rapidamente no mercado.

## Acceptance Criteria

### AC-1: Rule evaluator
- [ ] Loads active alert rules from DB
- [ ] Evaluates rules against incoming market events
- [ ] Supports all AlertType conditions: funding_rate, liquidation_size, oi_change_pct, whale_trade, price_change
- [ ] Respects cooldown_min from alert_rules

### AC-2: Alert dispatcher
- [ ] Sends alerts via Telegram Bot API (`/sendMessage`)
- [ ] Rich alert format: symbol, exchange, side, value, mini context
- [ ] Telegram message uses HTML parse_mode for formatting

### AC-3: Rule cache
- [ ] In-memory cache of active rules
- [ ] Refreshes from DB every 60 seconds
- [ ] Grouped by alert type for fast lookup

### AC-4: Redis integration
- [ ] Connects to Redis PUB/SUB channels: cv:funding, cv:liquidations, cv:oi
- [ ] Parses events and routes to rule evaluator

### AC-5: Telegram verification (deep linking)
- [ ] Bot verifies users via /start AUTH_TOKEN deep link approach
- [ ] Stores verified chat_id in alert_channels.config

### AC-6: Wiring
- [ ] Alert engine wired up in `apps/alerts/src/index.ts`
- [ ] Graceful startup and shutdown

### AC-7: Tests
- [ ] Test: Rule evaluator matches conditions correctly
- [ ] Test: Cooldown enforcement prevents duplicate alerts
- [ ] Test: Rich alert format includes all required fields
- [ ] Test: Rule cache refreshes correctly

## Dev Notes

- Alert engine builds on existing Sentry setup in apps/alerts/src/index.ts
- Redis PUB/SUB channels already published by ingestion worker
- Telegram Bot API: `https://api.telegram.org/bot{TOKEN}/sendMessage`
- Rule cache is in-memory with 60s DB refresh (eng review decision)
- Use `@upstash/redis` for PUB/SUB client
