# S2B-4: Tier Gating Middleware

**Epic:** Sprint 2 Expandida — Parte B (Ciclo de Venda)
**Status:** Draft
**Type:** Backend
**Effort:** S (1 dia)
**Priority:** P1
**Depends on:** S2B-1 (auth), S2B-2 (plan field populated)

## Story

Como sistema CryptoVision, preciso que endpoints de API e features do dashboard respeitem o tier do usuário, para que free users vejam dados delayed e paid users acessem real-time e features avançadas.

## Acceptance Criteria

### AC-1: API tier gating
- [ ] All `/api/v1/auth/*` endpoints require authentication + check user tier
- [ ] `tierGateResponse()` returns 403 with upgrade_url when tier insufficient
- [ ] Free tier: only delayed data (existing public endpoints)
- [ ] Lite+: real-time data access via authenticated endpoints
- [ ] Starter+: API key generation, custom alerts
- [ ] Pro+: cross-exchange comparison, webhooks

### AC-2: Feature flags in frontend
- [ ] Hook `useTier()` returns current user's plan and feature access
- [ ] Components check tier before rendering premium features
- [ ] Upgrade prompt shown in place of locked features (with CTA to /pricing)

### AC-3: Rate limiting per tier
- [ ] Rate limiter uses user's plan to determine limit (from TIER_LIMITS.apiReqPerDay)
- [ ] Authenticated requests identified by user ID
- [ ] Unauthenticated requests: free tier limits by IP
- [ ] 429 response with Retry-After header when exceeded

### AC-4: Tests
- [ ] Teste: Free user gets 403 on premium endpoint
- [ ] Teste: Paid user accesses premium endpoint
- [ ] Teste: Rate limit enforced per tier
- [ ] Teste: useTier hook returns correct access flags
